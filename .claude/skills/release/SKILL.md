# Ink Monitor — Release

Walks through cutting a SemVer release of Ink Monitor. The full process
is documented in [`RELEASE.md`](../../RELEASE.md); this skill is the
condensed, runnable form.

## When to invoke

The user says "cut a release", "tag v0.x.0", "ship a new version", or
runs the skill with `patch|minor|major` arg.

## Steps the skill executes

### 1. Pre-flight

```bash
git checkout main
git pull --rebase
git status        # must be clean
git log --oneline -5
```

If dirty: stop and ask the user. If behind: rebase.

### 2. Bump version (default: minor)

```bash
npm version <patch|minor|major>
```

`npm version` does:
1. Updates `package.json` version
2. Creates a git commit
3. Creates a git tag

Verify the tag:
```bash
git tag -l | tail -3
git log --oneline -1   # shows the version bump commit
```

### 3. Update CHANGELOG.md

Open `CHANGELOG.md`. The structure is:

```markdown
## [Unreleased]

### Added
- ...

## [X.Y.Z] — YYYY-MM-DD

### Added
- ...
```

The skill should:
1. Move all `[Unreleased]` items into a new `[X.Y.Z] — YYYY-MM-DD]` section
2. Leave a fresh, empty `[Unreleased]` section at the top
3. Add a diff URL at the bottom:
   ```markdown
   [X.Y.Z]: https://github.com/YiJing233/ink-monitor/compare/v<prev>...vX.Y.Z
   ```
4. Commit: `git add CHANGELOG.md && git commit -m "docs: changelog for vX.Y.Z"`

### 4. Push

```bash
git push
git push --tags
```

### 5. GitHub release

```bash
gh release create vX.Y.Z \
  --title "vX.Y.Z" \
  --notes-file /tmp/release-notes-vX.Y.Z.md
```

The release notes file should be generated by the skill from the
CHANGELOG section. A template is in `RELEASE.md` — copy + fill.

If the user has not yet authenticated `gh`, stop and tell them to run
`gh auth login` in a terminal.

### 6. Verify

```bash
gh release view vX.Y.Z --web   # opens the release page in a browser
```

Confirm:
- Title is `vX.Y.Z`
- Body has the Highlights / What's new / Bug fixes sections
- The tag is marked "Latest" (or "Pre-release" for RC)

### 7. Deploy (if main auto-deploys to Vercel)

If Vercel is configured to deploy `main` automatically (the default
with the vercel.json in this repo), the production site is already
updated. Verify:

```bash
curl -fsS https://<production-domain>/api/health
# should return version "X.Y.Z" in the body
```

If Vercel is configured to deploy on tags only, run `npx vercel --prod`.

### 8. Post-release comms

Print the release URL and the Twitter / Mastodon / newsletter template
text:

```
🚀 Ink Monitor vX.Y.Z is out

<one-line highlight>

<second-line highlight>

→ https://github.com/YiJing233/ink-monitor/releases/tag/vX.Y.Z
```

## Common issues

- **`npm version` fails with "not a git repository"**: the skill was
  invoked in the wrong dir. `cd` into the project root first.
- **Tag already exists**: someone pushed a tag you don't know about.
  `git fetch --tags` and check, then bump patch to a new number.
- **CHANGELOG has merge conflicts**: the skill should not be run on
  a branch with unmerged changes. Cherry-pick the CHANGELOG edit
  manually.

## Hotfix variant

For a one-off security fix that can't wait for a regular release:

```bash
git checkout main
git pull
git checkout -b hotfix/fix-name
# fix
git commit -m "fix: ..."
npm version patch
git push && git push --tags
gh release create vX.Y.Z+1 --title "vX.Y.Z+1 (hotfix)" --notes "..."
git checkout main
git merge hotfix/fix-name --ff-only
git push
```

Skip the `npm version minor`/`major` arg — patch is the only sane
choice for a hotfix.
